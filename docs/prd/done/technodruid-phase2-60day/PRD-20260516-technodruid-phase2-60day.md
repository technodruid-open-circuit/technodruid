- **Reflection:** Four carries.
  - (1) **Brainstorm-spec-plan-execute as a workflow for doctrinal content.** This iter chained four skills end-to-end: `brainstorming` (6 doctrinal positions surfaced, walked one-at-a-time with multi-choice format, approved); `writing-plans` (15-task plan with concrete prose per section drawn from the spec's content sketches); `executing-plans` (mechanical task execution); and the project's native `prd_work_loop` invariant pre-flight + VERIFY pattern as the inner verification machinery of the executing-plans tasks. The combined cost: roughly 90 minutes of user-facing brainstorm + 15 minutes of plan writing + 25 minutes of execution + verification. The alternative would have been "ask the user to draft the Charter body" — which would have left the doctrinal positions implicit, ungrounded, and unreviewable. The brainstorm front-loaded the positions; the plan front-loaded the prose; the execution was mechanical. The pattern generalizes to any future canonical drafting: positions before prose, prose before execution, execution before verification, verification before commit. Each stage can be reviewed and corrected at its own granularity.
  - (2) **Spec-first paid out at exactly the predicted scale.** The spec's per-section content sketches read like bullet-form prose. The draft put the bullets in declarative third-person and added RFC 2119 binding language. There was zero substantive prose-vs-spec drift detected by the Task 10 spot-checks (the one "founding ceremony" match was a benign negation). The 115-line Charter is shorter than the 150-200 target because the spec did the structural work; the draft mostly translated. **Predicted cost of this iter going in: ~30-45 min.** Actual: ~25 min. Below estimate. Carry forward: when a doctrinal spec is approved at the level of detail this one was, the drafting iteration's cost is dominated by VERIFY, not by ACT.
  - (3) **The "false positive in negative-evidence grep" pattern, generalized.** Iter 16's reflection (2) said: "when verifying a `must not have X` invariant against a file with by-design exceptions, the pre-flight checklist item should include 'expected exempt matches'." This iter's Task 10.5 demonstrated the *other* direction of the same hazard: a negative-evidence grep can fire on a positive doctrinal use (saying "X is NOT required") that contains the X-pattern textually. Both directions need the same discipline — enumerate, at pre-flight, what counts as a benign match and why. This iter, I caught it at VERIFY rather than at pre-flight, which worked but cost extra read-the-context time. Carry forward: when authoring a negative-evidence grep check in a plan or pre-flight, also predict the benign-match patterns and document them alongside the expected zero-match expectation.
  - (4) **The RFC 2119 legend up front was the right call.** A reader who scans the Charter top-to-bottom encounters the legend before the first MUST in §1. Without the legend, the first MUST would have demanded interpretation; with it, the binding language is calibrated before use. The legend is two sentences and pays for its own weight by every subsequent MUST/SHOULD/MAY it qualifies. This isn't novel insight — it's the RFC 2119 community's own practice — but it's worth surfacing as a project-specific carry: future canonical docs that use binding language SHOULD declare the binding-language semantics at the top of the document, not in a glossary or footer.
